# ЦУП KSP2 — полноценный командный центр с ЛЛМ (план реализации)

## Context

Сейчас есть работающая связка: мод KSP2 (C#) → TCP-мост → backend (FastAPI, шина Hub в
GMSEC-стиле, один агент FLIGHT, инструменты управления, SQLite-журнал) → два фронта:
кастомный пульт `:3000` (React/CSS) и настоящий Open MCT `:3001` с готовыми дашбордами.

Проблемы, которые решает этот план:
- Кастомный фронт выглядит «не как настоящий ЦУП» и дублирует роль Open MCT.
- Управление ведёт один агент — нет разделения по операторским позициям, нет
  формального контроля готовности (GO/NO-GO), лётных правил, циклограммы и фаз полёта.
- Баллистика/манёвры считаются «на глаз» внутри ЛЛМ — нет точного расчёта.
- Нет 3D «большого экрана» ситуации.

Цель: собрать максимально достоверный ЦУП по образцу реального управления полётами,
задействовав ПО из каталога NASA. Утверждённые решения:
**Astro UXDS** для внешнего вида, **группа операторов** (мультиагент), **GMAT** для
баллистики, **CesiumJS** для 3D-табло.

## Что берём из NASA / реального ЦУП (обосновано ресёрчем)

| Компонент | Роль в проекте | Источник |
|---|---|---|
| **Open MCT** (уже есть) | Стена телеметрии: графики, приборы, LAD, дашборды по фазам | software.nasa.gov ARC-15256; используется на InSight, MarCO, Cold Atom Lab |
| **GMSEC API / C2MS** (частично есть) | Шина сообщений: стандартные типы HB/TLM/MVAL/CMD/LOG, иерархические subject | software.nasa.gov GSC-19090; стандарт OMG C2MS |
| **GMAT** | Баллистический центр (FDO): расчёт манёвров, выведения, сближения, таргетинг | software.nasa.gov GSC-19640; Python API; операционно на TESS, ACE |
| **Astro UXDS** | Дизайн-система командного фронта (часы, таймлайны, статус-иконки, журнал, тревоги) | Open-source Space Force/SSC (R2C2/SATOPS); `@astrouxds/react` v7 |
| **CesiumJS** | 3D-глобус Kerbin с орбитой/трассой/аппаратом/целью | Преемник закрытого NASA WebWorldWind |

Операционная модель — по реальному MCC: позиции операторов (FLIGHT/ГОГУ, FIDO/динамика,
BOOSTER/ДУ, EECOM/СЭП-СОТР, GNC/СУ, CAPCOM/связь), предстартовый опрос GO/NO-GO по
«голосовым петлям», лётные правила (flight rules), фазы полёта и циклограмма с
предстартовым отсчётом и встроенными задержками (built-in holds).

## Целевая архитектура (сервисы)

```
mod (KSP2, C#) --TCP:8766--> backend(FastAPI:8000)
   backend: bridge · telemetry · Hub[GMSEC/C2MS] · mission-clock+phases ·
            flight-rules-engine · FLIGHT + позиции(FIDO/BOOSTER/EECOM/GNC/CAPCOM) ·
            procedures · tools · SQLite
   backend <--HTTP--> gmat-service(:8100)   (баллистика, Python API GMAT, система Kerbol)
   фронты:
     :3000  Astro UXDS  — командный пульт (директивы ЛЛМ, GO/NO-GO, циклограмма,
                          журнал/петли, стыковка, ручное, вкладка баллистики)
     :3001  Open MCT     — телеметрийная стена (дашборды, condition sets, LAD, notebook)
     :3002  CesiumJS     — 3D ситуационный экран (можно встраивать в :3000)
```

## Фазы реализации

Порядок выбран так, чтобы каждый слой опирался на предыдущий. Рекомендуемый первый
рубеж — Фазы 0–1 (операционный костяк + мультиагент): именно они делают систему «ЦУП».

### Фаза 0 — Операционный костяк: время, фазы, лётные правила (backend)
Единый источник истины о состоянии миссии, из которого питаются и UI, и агенты.
- Новый `backend/app/mission.py`: миссионное время (MET), предстартовый отсчёт (T‑), и
  конечный автомат фаз: `PRELAUNCH → ASCENT → ORBIT → RENDEZVOUS → DOCKING → DEORBIT →
  ENTRY → LANDING → SAFED`. Публикует `mission_state` на шину; фаза выводится из
  телеметрии (ситуация/высота/скорость/двигатели) + ручное переключение.
- Новый `backend/app/flight_rules.yaml` + `backend/app/rules.py` — декларативный движок
  лётных правил: пороги caution/warning и условия GO/NO-GO по позициям, привязанные к
  фазам. **Консолидирует разрозненную логику** из
  `frontend/src/components/CautionWarning.tsx` и `ReadinessPoll.tsx` в один авторитетный
  источник на backend. Отдаёт: REST `/api/rules/state`, события шины (caution/warning,
  poll), и инструмент ЛЛМ `check_flight_rules`.
- Тонкость: пороги для Kerbin (атмосфера 70 км, орбитальная ~2250 м/с) вынести в конфиг.

### Фаза 1 — Группа операторов: мультиагент (backend)
- Рефактор `backend/app/agent.py`: `DispatcherAgent` остаётся как **FLIGHT** (руководитель,
  оркестратор). Новый `backend/app/positions.py` — операторские позиции:
  `FIDO` (динамика/траектория), `BOOSTER` (ДУ/топливо), `EECOM` (СЭП/СОТР),
  `GNC` (ориентация/СУ), `CAPCOM` (связь/экипаж). У каждой — свой срез телеметрии и свой
  набор лётных правил (из Фазы 0).
- Экономия токенов: позиции **сначала на правилах** (детерминированный GO/NO-GO из
  движка), ЛЛМ-рассуждение позиции вызывается только по опросу или аномалии. Возможен
  более дешёвый/локальный model для позиций (`MCC_POSITION_MODEL`).
- Оркестрация: у FLIGHT новые инструменты `poll_positions` (собрать GO/NO-GO со всех
  позиций) и `ask_position(name, question)` (делегировать анализ). Доклады позиций
  публикуются на шину (см. Фаза 2). FLIGHT интегрирует и принимает решение — как реальный
  предстартовый опрос по петлям.
- Тонкость: 7B локально не «летает» в быстрых фазах — тайм-критичное держат правила +
  автопилот (Harmony attitude в моде), ЛЛМ ведёт надзор/решения. Сохранить текущую логику
  коротких `wait()` из системного промпта.

### Фаза 2 — GMSEC / C2MS: формализация шины (backend)
- Расширить `backend/app/gmsec.py` до типов C2MS: `MSG.HB` (есть), `MSG.TLM`,
  `MSG.MVAL` (mnemonic value — значение параметра со статусом лимита), `REQ.DIR`/`RESP.DIR`
  (директивы/ответы), `MSG.LOG`, `MSG.CMD`. Каждой позиции — свой компонент/subject
  (`KSP2.FIDO.MSG.*` и т.д.). Задокументировать таксономию subject в `README`.
- Опционально (не блокер): интерфейс внешней шины (NATS/MQTT) за `Hub`, если позиции
  вынести в отдельные процессы. По умолчанию — текущий in-process `Hub`.

### Фаза 3 — Баллистический центр: сервис GMAT (новый сервис)
- Новый `gmat/` (Docker): образ с GMAT R2022a+ и его Python API (`load_gmat`/`gmatpy`),
  обёрнут тонким FastAPI на `:8100`.
- **Определить систему Kerbol как кастомные тела GMAT** (SolarSystem/CelestialBody):
  Kerbin (GM, радиус, атмосфера, SOI), Mun, Minmus, Kerbol — константы из KSP2. Без этого
  расчёты будут «земные» и неверные — критическая тонкость.
- Эндпоинты: `/propagate`, `/plan_transfer` (Хохманн/би-эллипс до заданных Ha/Hp),
  `/plan_plane_change`, `/plan_rendezvous` (фазировка), `/targeting` (диффкоррекция).
  Возвращают жжения: `ut` + `dv{prograde,normal,radial}` → backend превращает в узлы
  манёвра KSP2 существующим путём (`CommandProcessor.CreateNode` /
  `Maneuvers.AddNodeToVessel`).
- Инструменты ЛЛМ: `plan_transfer`, `plan_rendezvous`, `propagate_orbit` (в
  `backend/app/tools.py`), ими пользуется в первую очередь позиция FIDO.
- Тонкости/риски: образ GMAT крупный (~сотни МБ–ГБ); Python API требует конкретной сборки
  GMAT — заложить время на сборку/проверку. Запасной вариант, если API GMAT окажется
  болезненным: `Orekit`/`poliastro` с теми же Kerbol-константами (тот же контракт REST,
  фронт/агенты не меняются).

### Фаза 4 — Командный фронт на Astro UXDS (:3000)
- Переписать `frontend` на `@astrouxds/react` (v7, зафиксировать версию — контракт
  завершён в окт 2025, но пакет стабилен в npm). Карта компонентов:
  - `rux-global-status-bar` — верхняя панель (миссия, маркировка, связь, статус шины GMSEC)
  - `rux-clock` ×N — UTC, MET, отсчёт T‑, TIG узла манёвра
  - `rux-monitoring-icon` / `rux-status` — табло GO/NO-GO и здоровье подсистем (из правил)
  - `rux-log` — журнал операций в стиле «петель» (из `event_log`/шины)
  - `rux-notification` / `rux-push-button` — мастер-тревога и квитирование; органы управления
  - `rux-timeline` — предстартовый отсчёт и циклограмма/процедуры
  - `rux-segmented-button`/`rux-tabs` — вкладки пульт/стыковка/ручное/баллистика
  - `rux-table` — LAD и таблица опроса позиций
- Сохранить бизнес-логику: reducer/websocket в `frontend/src/App.tsx`, `ws.ts`, `types.ts`
  переиспользовать как есть — меняем только представление на Astro. Диалог с ЛЛМ
  (`Console.tsx`), стыковка (`DockingDisplay.tsx`), ручное (`ManualPanel.tsx`) —
  перенести под Astro-обёртку. Плитки-ссылки на Open MCT (:3001) и 3D (:3002) оставить.

### Фаза 5 — 3D ситуационный экран (CesiumJS, :3002)
- Новый сервис/страница CesiumJS: глобус с **кастомным эллипсоидом Kerbin** (радиус,
  без Ion-токена, локальная текстура — офлайн), сущность аппарата, линия орбиты (из
  элементов орбиты телеметрии), трасса полёта, метка цели. Данные — по WS `:8000/ws`.
- Встраивается в Astro-фронт как отдельная вкладка/окно.
- Тонкость: Cesium по умолчанию — Земля WGS84 и внешние тайлы Ion; переопределить на
  Kerbin и локальные ассеты, чтобы работало офлайн и в масштабе игры.

### Фаза 6 — Полировка Open MCT (телеметрийная стена)
- Condition Sets для подсветки по лимитам (те же пороги, что в flight_rules), LAD Tables
  на дашбордах, плагин Notebook для бортжурнала, доводка геометрии/заголовков плиток,
  дашборды под каждую фазу. Пробросить статус лимита в метаданные телеметрии (enumerations/
  limits), чтобы Open MCT красил значения штатно. Устраняет «не красиво» и на стороне :3001.

### Фаза 7 — Процедуры/чек-листы и циклограмма (backend + UI)
- Декларативные процедуры (YAML): выведение, гравповорот, циркуляризация, сближение,
  стыковка, сход с орбиты, вход, посадка. Движок ведёт FLIGHT по шагам; UI показывает
  прогресс чек-листа (Astro) + таймлайн. Фаза → активные правила → активные позиции.
- (Опц.) доработки мода: телеметрия связи/антенн, потери сигнала (LOS/AOS), потоки EC,
  тепловой режим по частям — для достоверных докладов EECOM/CAPCOM.

## Критические тонкости и риски (свести в одном месте)
- **GMAT ↔ KSP2**: обязательно определить тела Kerbol с игровыми константами, иначе узлы
  манёвров неверны. Образ тяжёлый; Python API капризен → запасной Orekit/poliastro с тем
  же REST-контрактом.
- **Мультиагент и стоимость**: позиции детерминированы на правилах, ЛЛМ — по запросу; для
  позиций можно дешёвую модель. Тайм-критичное — правила+автопилот, не ЛЛМ.
- **Astro UXDS v7 EOL**: пакет стабилен, но не развивается — зафиксировать версию, не
  завязываться на будущие обновления.
- **CesiumJS**: без Ion-токена, кастомный Kerbin, локальные ассеты (офлайн, масштаб игры).
- **Open MCT CSP/asset-path**: уже решено (Conductor + `<div id=app>` + `DOMContentLoaded`),
  не регрессировать.
- **Секрет в `.env`** (ключ Cloud.ru) — не логировать, не коммитить; продолжать бережно.
- **Реальное время vs латентность ЛЛМ** — надзорная роль ЛЛМ, исполнение критичного —
  правила/автопилот; сохранить короткие `wait()`.

## Проверка (end-to-end)
1. `docker compose up -d` поднимает: backend `:8000`, openmct `:3001`, astro `:3000`,
   cesium `:3002`, gmat `:8100`. Проверить `/api/status`, `/api/rules/state`,
   `GET :8100/health`.
2. Фаза 0/1: без игры — `GET /api/rules/state` даёт «нет данных»; запустить KSP2 с модом,
   на старте `poll_positions` возвращает GO/NO-GO по позициям; фаза = PRELAUNCH→ASCENT
   при взлёте.
3. Фаза 3: `POST :8100/plan_transfer` с текущей орбитой → возвращает жжение; директива
   «подними апогей до 200 км» → FIDO зовёт GMAT → в игре появляется корректный узел →
   FLIGHT исполняет; итоговая орбита совпадает в пределах допуска.
4. Фаза 4: `:3000` рендерит Astro-пульт (часы, GO/NO-GO табло, журнал-петли, тревога),
   директива уходит и стримится ответ; вкладки стыковка/ручное работают.
5. Фаза 5: `:3002` показывает Kerbin, орбиту и метку аппарата, обновляется в реальном
   времени; трасса совпадает с трассой Open MCT.
6. Фаза 6: значения в Open MCT краснеют при выходе за лимит (condition sets = flight_rules).
7. Сценарий целиком: «выведи на орбиту 100 км, найди цель, сблизься и состыкуйся» —
   предстартовый GO/NO-GO, выведение по циклограмме, FIDO+GMAT планирует круговую орбиту и
   сближение, стыковка по RCS; журнал и таймлайн отражают ход; 3D показывает сближение.

## Затрагиваемые файлы (ориентир)
- backend новые: `mission.py`, `rules.py`, `flight_rules.yaml`, `positions.py`, `procedures/`.
- backend правки: `agent.py` (FLIGHT-оркестратор), `tools.py` (poll_positions, ask_position,
  plan_transfer/rendezvous/propagate, check_flight_rules), `gmsec.py` (типы C2MS),
  `main.py` (запуск mission/positions, новые REST), `config.py` (модели/пороги/URL GMAT).
- новый сервис `gmat/` (Dockerfile, app FastAPI, определение системы Kerbol).
- фронт `frontend/` — переезд на Astro UXDS (переиспользуя App-reducer/ws/types).
- новый сервис `cesium/` (:3002) или вкладка в Astro-фронте.
- `openmct/public/*` — condition sets, LAD, notebook, доводка дашбордов.
- `docker-compose.yml` — сервисы gmat, cesium (и astro вместо текущего frontend).
